# dstack v0.1 — Design Spec

**Date:** 2026-04-08
**Author:** bkataru
**Status:** APPROVED

## Overview

dstack is an open-source Claude Code plugin + Rust CLI that provides persistent memory, multi-repo orchestration, and production deployment skills for AI-assisted development. Built from 2 weeks of real production work across 15+ repos at DIRMACS.

## Design Decisions

1. **Audience:** Open-source. Public core with private overlay for DIRMACS-specific config (paths, secrets, endpoints). No proprietary info in the public repo.

2. **Memory layer:** Pluggable `MemoryProvider` trait with two shipped backends:
   - `FileProvider` (default) — JSON files in `~/.dstack/memory/`
   - `ErukaProvider` — REST API calls to Eruka context engine
   - Third parties can implement Redis, SQLite, etc.

3. **Relationship to superpowers/GSD:** Standalone but compatible. No skill name collisions. Does NOT reimplement TDD, debugging, wave execution — lets superpowers and GSD handle those. dstack focuses on what they lack: persistent memory, multi-repo ops, VPS deployment, companion docs, quality gates.

4. **Scope for v0.1:** 5+1 skills, 3 CLI command groups, 3 hooks, config system. Minimal viable plugin that ships unique value.

5. **Architecture:** Hybrid — Rust CLI binary (`dstack`) for memory/sync/deploy logic + Claude Code plugin (skills/hooks/commands as markdown) for agent behavior. Commands in the plugin are thin wrappers calling the CLI.

## CLI Commands

```
dstack memory load [--project <name>]     # Load context for session
dstack memory save <key> <value>          # Persist a learning
dstack memory query <pattern>             # Search memory fields
dstack memory export                      # Dump all as JSON

dstack deploy <service>                   # Build + restart + smoke test
dstack deploy --all                       # All services in dep order

dstack sync [--dry-run]                   # Git sync all tracked repos
dstack sync --status                      # Dirty/ahead/behind per repo

dstack audit [--pre-commit]               # 5-question quality gate
dstack audit --stale                      # Find stale docs
dstack audit --coverage                   # Test coverage gaps

dstack context-monitor                    # Token usage heuristic check

dstack init                               # Set up .dstack/ in a project
dstack config                             # Show/edit config
```

## Configuration

`~/.config/dstack/config.toml`:

```toml
[memory]
backend = "file"                          # "file" or "eruka"
path = "~/.dstack/memory"

[memory.eruka]
url = "http://localhost:8081"
service_key = ""                          # or $DSTACK_ERUKA_KEY

[repos]
root = "/opt"
tracked = ["ehb", "eruka", "ares", "dirmacs-core"]

[deploy.ehb]
```

## MemoryProvider Trait

```rust
#[async_trait]
pub trait MemoryProvider: Send + Sync {
    async fn load(&self, path: &str) -> Result<Vec<Field>>;
    async fn write(&self, field: &Field) -> Result<()>;
    async fn search(&self, query: &str) -> Result<Vec<Field>>;
    async fn delete(&self, path: &str) -> Result<()>;
}

pub struct Field {
    pub path: String,
    pub value: String,
    pub confidence: f64,
    pub source: String,
    pub updated_at: DateTime<Utc>,
}
```

**FileProvider:** JSON files at `{config.memory.path}/{path}.json`. Search via substring match over values. Zero dependencies.

**ErukaProvider:** Maps to `GET/POST /api/v1/context` with `X-Service-Key` header. Field ↔ Eruka field 1:1 mapping.

## Skills (6 total)

### 0. using-dstack
**Trigger:** Session start
**Purpose:** Explain dstack, show available commands, confirm memory loaded.

### 1. eruka-memory
**Trigger:** Any session where dstack is active
**Purpose:** Guide when/how to persist learnings. "If you discover something non-obvious, run `dstack memory save`." Memory hygiene: don't save ephemeral state, DO save corrections, patterns, architectural decisions.

### 2. multi-repo-ops
**Trigger:** Work touches 2+ repos in tracked list
**Purpose:** Cross-repo dependency awareness. Build order: dirmacs-core → ares → ares-dirmacs → ehb. Test chain. Sync protocol. Branch alignment (holy↔main).

### 3. vps-deploy
**Trigger:** After release build or user says "deploy"
**Purpose:** `dstack deploy <service>` workflow: build → restart → smoke test → rollback on failure. Service-aware via config.toml.

### 4. companion-docs
**Trigger:** Creating plans or finishing features
**Purpose:** Every plan gets `.implementation.md`. Track reality vs intent. Update status on completion. Don't let docs rot.

### 5. quality-gates
**Trigger:** Before any commit (via PreCommit hook)
**Purpose:** 5-question checklist:
1. Did I write negative tests?
2. Did I verify against the LIVE system?
3. Did I update the companion doc with DETAILS?
4. Would these tests FAIL without my code change?
5. Am I moving to the next step because current is TRULY done?

## Hooks (3 total)

### SessionStart
Calls `dstack memory load --project auto`. Detects project from cwd, loads relevant memory fields, injects as context.

### PreCommit
Calls `dstack audit --pre-commit`. Runs quality gate. Can block commit in strict mode.

### PostToolUse (context-monitor)
Async. Tracks tool call count as proxy for context usage. Warns when context is heavy, suggests saving important findings.

## Compatibility

- **Superpowers:** No conflicts. dstack skills use `dstack:` prefix. Superpowers handles TDD, debugging, verification.
- **GSD:** No conflicts. GSD handles project structure, waves, milestones. dstack handles memory, deploy, multi-repo.
- **Both installed:** Users get the full stack: GSD for structure → superpowers for execution → dstack for memory + ops.

## Security

- No secrets in the public repo
- `config.toml` is user-local (`~/.config/dstack/`)
- `overlays/` directory has `.gitignore` example showing how to layer private config
- ErukaProvider reads service key from env var `$DSTACK_ERUKA_KEY`, never from files in repo
- No telemetry, no phone-home

## Out of Scope (v0.1)

- Wave execution engine (GSD handles this)
- TDD enforcement (superpowers handles this)
- NIM model triage (future skill)
- pawan delegation protocol (future skill)
- Plugin marketplace submission (after dogfooding)
- Multi-LLM delegation (Night Market's conjure concept — future)
